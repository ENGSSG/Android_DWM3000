# DWM3000 Tracker — End-to-End System Overview

Pixel phone (AndroidX UWB) ↔ nRF52840DK + DWM3000 firmware. BLE is used as a
one-shot Out-Of-Band (OOB) channel to hand off FiRa session parameters; UWB
ranging then runs entirely on the radios with no further BLE traffic.

```
┌──────────────────────┐                       ┌──────────────────────┐
│  Pixel phone         │                       │  nRF52840DK +        │
│  (DWM3000 Tracker)   │                       │  DWM3000             │
│                      │   BLE GATT (FFF0)     │                      │
│  BLE Central         │  ◄──────────────────► │  BLE Peripheral      │
│  UWB Controller      │  16-byte FFF1 write   │  UWB Controlee       │
│  AndroidX UWB stack  │   2-byte FFF2 notify  │  Qorvo FiRa stack    │
│                      │                       │                      │
│                      │   UWB DS-TWR rounds   │                      │
│  Pixel UWB modem     │  ◄═════════════════►  │  DWM3000 transceiver │
│                      │   FiRa MAC v2.0       │                      │
└──────────────────────┘                       └──────────────────────┘
```

## 1. Roles and stacks

| | Phone | Board |
|---|---|---|
| BLE role | Central (GATT client) | Peripheral (GATT server, advertises FFF0) |
| UWB role | Controller / Initiator | Controlee / Responder |
| BLE stack | Android Bluetooth (`BluetoothGatt`) via `BleCentral.kt` | Nordic SoftDevice S140 via `Src/Comm/Src/BLE/nrfx/ble.c` + `ble_fff0.{c,h}` |
| UWB stack | AndroidX UWB `1.0.0-alpha08` (Pixel modem under the hood) | Qorvo `uwbstack` (`fira_helper`) on FreeRTOS |
| App entry | `MainActivity.kt` | `Src/Apps/Src/ble_session/ble_session.c` |
| Ranging profile | `CONFIG_MULTICAST_DS_TWR` (CONFIG_ID_2), `RANGING_UPDATE_RATE_FREQUENT` | Mirrors the same profile via `apply_params_to_fira()` + helper TLV setters |

## 2. End-to-end timeline

1. **Board boots**
   `ble_init()` brings up the SoftDevice, registers the FFF0 service, and
   starts undirected advertising. The board sits in USB-UCI host-responder
   mode by default — UCI is alive, FiRa is dormant.
2. **Phone scans**
   The Controller path in the app calls
   `UwbRangingManager.prepareControllerScope()` to obtain the system-assigned
   UWB complex channel + preamble index + local 2-byte UWB address.
   `BleCentral.scanAndConnect()` then scans for an advertiser carrying the
   `FFF0` service UUID.
3. **Phone connects**
   Standard GATT connect → service discovery → enables notifications on `FFF2`
   (the board's address-echo characteristic). On the board, the
   `BLE_GAP_EVT_CONNECTED` handler latches the connection handle.
4. **Phone writes FFF1 (16 bytes)** — the OOB hand-off (see §3).
   Firmware fires `BLE_FFF0_EVT_PARAMS_WRITTEN`. The handler in `ble.c`:
   - Echoes the board's local UWB short address back via FFF2 (`ble_fff0_address_notify`).
   - Calls `ble_session_submit_params()` which decodes the payload and
     transitions UCI → FiRa Controlee (`uci_terminate` → `fira_uwb_mcps_init`
     → `fira_helper_open` → `fira_set_session_parameters` → BLES_SET TLVs →
     `uwbmac_start` → `fira_helper_start_session`).
5. **Phone reads FFF2 (2 bytes)** — the board's UWB short address arrives
   as a GATT notification. Phone now knows the peer address.
6. **Phone closes BLE.** AndroidX UWB calls `BluetoothGatt.close()` once the
   address is in hand. The board's `BLE_GAP_EVT_DISCONNECTED` handler
   *only* restarts advertising — it must NOT tear down the FiRa session
   (see INTEROP_NOTES §2).
7. **UWB ranging**
   Phone calls `UwbRangingManager.startAsController()` with the address
   it read from FFF2. Both modems run DS-TWR according to the pre-shared
   slot map. Each round produces a result frame (ToF + AoA) on the phone
   side and a `fira_notification_cb` callback on the board side
   (`ble_session: range peer=0x… d=Xcm`).
8. **Phone displays.** `MainActivity.handleRangingData()` updates the
   distance/azimuth/elevation HUD; if Stage 2 is enabled, the
   `ArOverlayView` reticle and `RadarView` mini-radar update too.

## 3. The OOB payload

Three GATT objects on a single custom service.

| UUID | Direction | Bytes | Purpose |
|---|---|---|---|
| `FFF0` (service) | — | — | Advertised by the board so the phone can find it. |
| `FFF1` (write) | Phone → board | 16 | UWB session parameters (phone-assigned). |
| `FFF2` (notify) | Board → phone | 2 | Board's local UWB short address. |

### FFF1 — 16 bytes, little-endian
Encoded by `UwbSessionParams.toByteArray()` in
`app/src/main/java/com/dwm3000/tracker/UwbSessionParams.kt`. Decoded by
`decode_oob_payload()` in `Src/Apps/Src/ble_session/ble_session.c`.

| Off | Len | Field | Notes |
|---:|---:|---|---|
| 0 | 4 | `sessionId` | FiRa Session ID, randomly generated by the phone. Both sides start the FiRa session with this value. |
| 4 | 1 | `channel` | UWB channel number (typically `9`). Phone reads it from `UwbControllerSessionScope.uwbComplexChannel.channel`. |
| 5 | 1 | `preambleIndex` | UWB preamble code index (typically `10`). Same source. |
| 6 | 8 | `sessionKey` | Static-STS key material. Controlee splits as `VENDOR_ID = sessionKey[0..1]`, `STATIC_STS_IV = sessionKey[2..7]`. **VID bytes go into `vupper64[6..7]` REVERSED** to match AOSP `isReversedByteOrderFiraParams=true`. |
| 14 | 2 | `peerAddress` | Phone's local UWB short address (= the board's destination). |

### FFF2 — 2 bytes
Single notification. Board returns `BLE_SESSION_LOCAL_SHORT_ADDR = 0x0001`
encoded little-endian as `01 00` (`m_local_uwb_addr` in `ble.c:41`). Phone
uses this to construct `UwbAddress` for the `peerDevices` list passed to
`RangingParameters`.

## 4. Pre-shared FiRa parameters (NOT in the OOB payload)

Both sides hardcode the topology and timing for the
`CONFIG_MULTICAST_DS_TWR` profile at `RANGING_UPDATE_RATE_FREQUENT`. Every
value below must match exactly — in `TIME_SCHEDULED + ONE_TO_MANY` mode
there is no per-round Ranging Control Message, so any disagreement causes
RX_TIMEOUT (status=33).

Sources of truth:
- Phone: AOSP `androidx.core.uwb.backend.impl.internal.ConfigurationManager`
  (CONFIG_ID_2 block) and `Utils.java` (`RangingTimingParams` for
  `CONFIG_MULTICAST_DS_TWR`).
- Board: `apply_params_to_fira()` and the BLES_SET block in
  `Src/Apps/Src/ble_session/ble_session.c`.

| Param | Value | Notes |
|---|---|---|
| `multi_node_mode` | `ONE_TO_MANY` (1) | Topology — even with one peer, AOSP uses ONE_TO_MANY for this profile. |
| `device_type / device_role` | `CONTROLEE / RESPONDER` (board), `CONTROLLER / INITIATOR` (phone) | Controller-is-initiator. |
| `ranging_round_usage` | `DS_TWR_DEFERRED` (2) | Double-sided two-way ranging, deferred mode. |
| `sts_config` | `STATIC` (0) | Static STS, key composed from `sessionKey`. |
| `rframe_config` | `SP3` (3) | STS-only rframe; no PSDU. |
| `schedule_mode` | `TIME_SCHEDULED` (1) | Both sides run a pre-shared slot map. |
| `block_duration_ms` | `120` | AOSP `rangingIntervalFast`. |
| `slot_duration_rstu` | `2400` | ~2 ms per slot. |
| `round_duration_slots` | `20` | AOSP `slotPerRangingRound` for CONFIG_ID_2. |
| `round_hopping` | `true` | AOSP `hoppingEnabled=true` → `HOPPING_MODE_FIRA_HOPPING_ENABLE`. |
| `mac_address_mode` | `2_BYTES` (0) | Short addressing. |
| `aoa_result_request` | `REQ_AOA_RESULTS` (1) | AoA requested. Pixel modem produces it (board has 1 antenna and cannot). |
| `result_report_config` | ToF + AoA azimuth + AoA elevation + AoA FoM | Bitfield: `0xF`. |
| `in_band_termination_attempt_count` | `3` | |
| `sts_length` | `64` | |
| `number_of_sts_segments` | `1` | |
| `preamble_duration` | `64` | |
| `psdu_data_rate` | `6M81` (0) | |
| `sfd_id` | `2` | |
| `key_rotation` | `0` | Static STS, no rotation. |

### V_UPPER_64 byte arrangement (Static STS)
Phone's AOSP UWB stack reverses the VID bytes before TLV transmit (because
`isReversedByteOrderFiraParams=true`). The Controlee must mirror that:

```
vupper64[0..5] = sessionKey[2..7]    // STATIC_STS_IV (in order)
vupper64[6]    = sessionKey[1]       // VID byte 1
vupper64[7]    = sessionKey[0]       // VID byte 0  (reversed)
```

Wrong arrangement → every round = `status=36 PHY_STS_FAILED`.

## 5. Phone-side flow (Kotlin)

Files in `Android_DWM3000/Android_DWM3000/app/src/main/java/com/dwm3000/tracker/`.

| File | Role |
|---|---|
| `ui/MainActivity.kt` | Permission gating, role buttons (Controller / Controlee), camera + AR toggle (`STAGE_2`), HUD updates. |
| `ble/BleCentral.kt` | Scans for FFF0 service, connects, writes FFF1, reads FFF2 notification, then `BluetoothGatt.close()`. |
| `ble/BlePeripheral.kt` | Advertises FFF0 (only used in phone-to-phone Controlee mode). |
| `ble/BleConstants.kt` | FFF0/FFF1/FFF2 UUIDs and `BLE_SESSION_PARAMS_LEN = 16`. |
| `UwbSessionParams.kt` | Encodes/decodes the 16-byte payload; generates random session ID + 8-byte sessionKey. |
| `uwb/UwbRangingManager.kt` | Wraps AndroidX UWB. `prepareControllerScope()` to grab channel/preamble/local-addr; `startAsController()` runs `RangingParameters(CONFIG_MULTICAST_DS_TWR, RANGING_UPDATE_RATE_FREQUENT, ...)` and emits `RangingData(distanceMeters, azimuthDegrees, elevationDegrees)` via callback. |
| `ui/ArOverlayView.kt` + `ui/AoAScreenMapper.kt` + `ui/CameraFovHelper.kt` + `ui/SensorOrientationManager.kt` + `ui/GravityRollSensor.kt` | Map (azimuth, elevation) + roll + camera FOV → screen coords for the AR reticle. |
| `ui/RadarView.kt` | Mini-radar drawing the peer relative to the phone heading. |

Controller-path sequence (the "tap Controller" flow):

```
prepareControllerScope()    → gets {channel, preamble, localAddr}
generate sessionId + sessionKey (random)
BleCentral.scanAndConnect(serviceUuid=FFF0)
BleCentral.writeFff1(payload = encode(sessionId, channel, preamble, sessionKey, localAddr))
BleCentral receives FFF2 notification → 2-byte boardAddr
BleCentral.close()
startAsController(sessionId, sessionKey, peerAddress=boardAddr) → RangingData callback
```

## 6. Board-side flow (C / FreeRTOS)

Files in `SDK/Firmware/Src/`.

| File | Role |
|---|---|
| `Comm/Src/BLE/nrfx/ble.c` | SoftDevice init, GAP, advertising, event router. Calls into `ble_fff0` and `ble_session`. |
| `Comm/Src/BLE/nrfx/ble_fff0.{c,h}` | Custom GATT service implementing FFF0/FFF1/FFF2. Emits `BLE_FFF0_EVT_PARAMS_WRITTEN` and `..._NOTIFY_ENABLED`. |
| `Apps/Src/ble_session/ble_session.{c,h}` | OOB param decode, FiRa session lifecycle, `vupper64` arrangement, all helper TLV setters, range-result callback. |
| `Apps/Src/fira/fira_app.c` + `fira_default_params.h` | FiRa stack defaults referenced by `apply_params_to_fira()`. |
| `Libs/uwbstack_libs/.../fira_helper.h` | Closed-source Qorvo helper API surface. |

Sequence on FFF1 write:

```
fff0_event_handler(BLE_FFF0_EVT_PARAMS_WRITTEN)
  → ble_fff0_address_notify(FFF2, m_local_uwb_addr)        // 2-byte echo
  → ble_session_submit_params(payload, conn_handle)
       → decode_oob_payload(payload, &p)
       → uci_terminate()                                    // free SPI/MAC
       → start_fira(&p):
            apply_params_to_fira(&p, &m_fira_param)         // struct init
            fira_uwb_mcps_init / fira_helper_open
            fira_helper_set_scheduler / prepare_measurement_sequence
            fira_helper_init_session
            QLOGI("cadence in: block=%ums slot=%uRSTU round_slots=%u", ...)  // probe
            fira_set_session_parameters
            BLES_SET(...)  // pin every TLV via fira_helper_set_session_*
            fira_helper_set_session_vendor_id / static_sts_iv
            uwbmac_start
            fira_helper_start_session
            QLOGI("FiRa Controlee active ...")
```

The `fira_notification_cb` registered in `fira_helper_open` then fires once
per round and emits the `ble_session: range peer=0x… d=Xcm | status=N`
RTT lines you see during operation.

## 7. Reverse path (phone-to-phone Controlee)

Selectable from the app's Controlee button. Phone advertises FFF0,
receives the FFF1 write from a peer Controller, returns its UWB short
address on FFF2, then runs `UwbRangingManager.startAsControlee()`. The
firmware path is unused in this mode. Slot schedule still has to match
between the two phones, but they both default to AOSP CONFIG_ID_2 so it
just works.

## 9. Per-section summary

- **§1 Roles** — Phone is BLE Central + UWB Controller/Initiator; board is
  BLE Peripheral + UWB Controlee/Responder.
- **§2 Timeline** — Boot → advertise → phone scans/connects → phone writes
  FFF1 → board echoes FFF2 → phone closes BLE → both modems range
  in-band.
- **§3 OOB payload** — A single 16-byte FFF1 write carries
  `sessionId|channel|preamble|sessionKey|peerAddr`; FFF2 returns the
  board's 2-byte short address.
- **§4 Pre-shared params** — Topology + timing + STS TLVs for
  `CONFIG_MULTICAST_DS_TWR` + `FREQUENT` are hardcoded identically on
  both sides; `vupper64` uses reversed-VID byte order to match AOSP.
- **§5 Phone flow** — `MainActivity` → `BleCentral` for OOB →
  `UwbRangingManager` runs AndroidX `RangingParameters` → HUD + AR
  overlay + radar consume the result stream.
- **§6 Board flow** — `ble.c`/`ble_fff0` deliver the FFF1 payload to
  `ble_session.c`, which terminates UCI and brings the FiRa Controlee up
  with the matching schedule, then streams range results via
  `fira_notification_cb`.
- **§7 Reverse path** — Phone-to-phone Controlee mode reuses the same
  GATT service in the opposite direction; firmware is not involved.
- **§8 Artefacts** — Known-good firmware hex + APK paths.

## 10. One-paragraph TL;DR

The phone is BLE Central and UWB Controller; the board is BLE Peripheral
and UWB Controlee. BLE is used only as a one-shot OOB channel: the phone
writes a 16-byte payload (FiRa session ID, UWB channel, preamble code,
8-byte Static-STS key, and its own 2-byte UWB address) to characteristic
FFF1, the board returns its own 2-byte UWB address as a notification on
FFF2, and BLE then closes. From that moment ranging runs purely on the
UWB radios using DS-TWR rounds scheduled in TIME_SCHEDULED ONE_TO_MANY
mode. Topology and timing are not in the OOB payload — both sides
hardcode the AOSP `CONFIG_MULTICAST_DS_TWR` defaults at
`RANGING_UPDATE_RATE_FREQUENT` (120 ms block, 2400 RSTU slot, 20
slots/round, hopping enabled, SP3 rframe, ToF+AoA reporting), and the
Controlee composes `V_UPPER_64` with the VENDOR_ID bytes reversed so the
Static-STS key matches the AOSP-default reversed-byte ordering.
